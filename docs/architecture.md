# هندسة النظام (Software Architecture)
## نظام ERP + CRM المتكامل

**الإصدار:** 1.0  
**التاريخ:** 2026-07-12  
**الفريق:** فريق التطوير

---

## 1. المبادئ العامة

- **الفصل التام (Separation of Concerns):** كل طبقة مسؤولة عن وظيفة محددة.
- **الوحداتية (Modularity):** النظام مقسم إلى وحدات مستقلة قابلة لإعادة الاستخدام.
- **التمركز حول البيانات (Data-Centric):** التدفق يبدأ من البيانات وينتهي بعرضها.
- **الأمان أولاً:** كل طبقة تطبق سياسات الأمان المناسبة.
- **الأداء:** تصميم يضمن سرعة التحميل والاستجابة.
- **التوافق مع PWA:** دعم العمل دون اتصال والمزامنة.

## 3. تفصيل الطبقات

### 3.1 طبقة العرض (Presentation Layer)
- **التكنولوجيا:** HTML5، CSS3، Vanilla JS (ES2023).
- **المكونات:** صفحات (Pages) ومكونات (Components) قابلة لإعادة الاستخدام (مثل: جدول، نموذج، بطاقة، تقويم، إلخ).
- **التنظيم:** كل صفحة لها مجلد خاص بها، وكل مكون له مجلد خاص به.
- **التفاعل:** يتم ربط الأحداث (Event Listeners) في ملفات JavaScript المنفصلة عن HTML.
- **التخزين المؤقت:** استخدام IndexedDB لتخزين البيانات التي يتم عرضها لتقليل طلبات الشبكة.
- **الاستجابة:** استخدام CSS Flexbox و Grid لتحقيق تصميم متجاوب بالكامل.

### 3.2 طبقة منطق الأعمال (Business Logic Layer)
- **التنظيم:** وحدات (Modules) مستقلة، كل وحدة تتوافق مع كيان (مثل: customers, orders, employees, إلخ).
- **الوظائف:** 
  - إدارة الحالة (State) لكل وحدة (مثل: قائمة العملاء، العميل الحالي، حالة التحميل، الأخطاء).
  - تطبيق قواعد الأعمال (مثل: حساب إجمالي الطلب، احتساب التأخير، تحديث الرصيد).
  - التحقق من الصلاحيات (Permission Checks) قبل تنفيذ أي إجراء.
  - التواصل مع طبقة الوصول إلى البيانات لإرسال واستقبال البيانات.
- **مدير الحالة (State Manager):** كائن مركزي يحتفظ بحالة التطبيق بأكمله، ويسمح للتغييرات بالانتشار إلى المكونات المعنية (نظام نشر/اشتراك بسيط).
- **الموجه (Router):** مسؤول عن تغيير الصفحات بناءً على عنوان URL، وتحميل الوحدات المناسبة، والتحقق من المصادقة.

### 3.3 طبقة الوصول إلى البيانات (Data Access Layer)
- **API Wrapper:** دوال JavaScript تستدعي نقاط نهاية Google Apps Script باستخدام `fetch`. تدير الطلبات، إعادة المحاولة، معالجة الأخطاء، وإضافة رؤوس المصادقة.
- **IndexedDB Wrapper:** دوال للتعامل مع IndexedDB (فتح قاعدة البيانات، إنشاء مخازن، إضافة، قراءة، تحديث، حذف). يتم استخدامها لتخزين البيانات محلياً للعمل دون اتصال.
- **مدير المزامنة (Sync Manager):** يراقب حالة الشبكة، ويقوم بمزامنة العمليات المعلقة (التي تم تنفيذها دون اتصال) مع الخادم عند استعادة الاتصال.

### 3.4 الطبقة الخلفية (Backend Layer) - Google Apps Script
- **البنية:** مشروع Google Apps Script واحد يحتوي على ملفات مقسمة حسب الوظيفة (مثل: `auth.gs`, `customers.gs`, `orders.gs`, إلخ).
- **نقاط النهاية:** يتم نشرها كـ Web App، وتستقبل طلبات HTTP (GET, POST, PUT, DELETE) مع مسار (Route) محدد في الرابط.
- **المصادقة:** تُستخدم خدمة `doGet` و `doPost` لمعالجة الطلبات، ويتم التحقق من JWT في رأس الطلب.
- **قاعدة البيانات:** التفاعل مع Google Sheets عبر `SpreadsheetApp`، حيث يتم تنفيذ عمليات CRUD باستخدام صفائف البيانات.
- **التخزين:** رفع وتنزيل الملفات باستخدام `DriveApp`.
- **التسجيل:** تسجيل جميع الطلبات والأخطاء في Sheet مخصص (Logs) مع الطابع الزمني وعنوان IP والمستخدم.

## 4. تدفق البيانات (Data Flow)

1. **طلب المستخدم:** يقوم المستخدم بالتفاعل مع الواجهة (مثل: النقر على زر "إضافة عميل").
2. **معالجة منطق الأعمال:** يتم استدعاء الوحدة المناسبة (مثل: `customers.addCustomer(data)`).
3. **التحقق:** تتحقق الوحدة من صحة البيانات، الصلاحيات، وتحديث الحالة المحلية.
4. **طلب API:** تقوم الوحدة باستدعاء طبقة الوصول إلى البيانات (`api.customers.create(data)`).
5. **التخزين المؤقت:** يتم حفظ البيانات في IndexedDB (إذا كان الوضع غير متصل مفعلاً) وإضافتها إلى قائمة المزامنة.
6. **طلب الخادم:** يتم إرسال طلب HTTP إلى Google Apps Script.
7. **معالجة الخادم:** يقوم Google Apps Script بالتحقق من المصادقة، الصلاحيات، التحقق من صحة البيانات، ثم تنفيذ العملية على Google Sheets.
8. **الرد:** يعيد الخادم استجابة (نجاح أو خطأ مع رسالة).
9. **تحديث الواجهة:** تستقبل الوحدة الرد، وتحدث الحالة المحلية، وتقوم بتحديث العرض (مثل: إعادة رسم الجدول، عرض إشعار نجاح).

---

## 5. إدارة الحالة (State Management)

- **حالة التطبيق:** كائن `AppState` يحتوي على:
  - `user`: بيانات المستخدم الحالي وصلاحياته.
  - `customers`: قائمة العملاء، العميل المحدد، حالة التحميل، الأخطاء.
  - `orders`: قائمة الطلبات، الطلب المحدد، إلخ.
  - ... لكل كيان مماثل.
- **تحديث الحالة:** يتم تحديث الحالة فقط من خلال دوال `setState` داخل الوحدات، والتي تقوم بإشعار المشتركين (المكونات) لإعادة العرض.
- **استمرارية الحالة:** يتم حفظ الحالة الأساسية في IndexedDB أو LocalStorage لاستعادتها عند إعادة تحميل الصفحة.

---

## 6. إدارة التنقل (Routing)

- **استراتيجية:** استخدام `hash-based routing` (مثل `#dashboard`, `#customers`, `#orders/123`).
- **الموجه:** يستمع إلى حدث `hashchange`، ويقرأ المسار، ويطابق مع الصفحات المسجلة، ويقوم بتحميل ملفات HTML و JavaScript الخاصة بالصفحة (تحميل كسول).
- **حماية المسارات:** قبل تحميل الصفحة، يتحقق الموجه من أن المستخدم مسجل الدخول ولديه الصلاحية للصفحة المطلوبة.

---

## 7. الوحدات (Modules)

كل وحدة هي مجلد يحتوي على:
- `module.js`: يحتوي على كل دوال المنطق الخاصة بهذه الوحدة (CRUD، حسابات، تحقق).
- `state.js`: يحتوي على تعريف الحالة الخاصة بالوحدة.
- `ui.js`: يحتوي على دوال إنشاء وتحديث عناصر واجهة المستخدم الخاصة بهذه الوحدة (يمكن استخدامها بواسطة الصفحات).
- `routes.js`: تعريف المسارات الفرعية (إذا كانت الوحدة تحتوي على صفحات فرعية مثل تفاصيل العميل).

---

## 8. الأمان

- **على العميل:** التحقق من صحة المدخلات، ترميز المخرجات لمنع XSS، استخدام HTTPS فقط، تخزين JWT في ذاكرة التطبيق (وليس localStorage لتجنب هجمات XSS) أو في Cookie آمن.
- **على الخادم:** التحقق من JWT في كل طلب، التحقق من الصلاحيات لكل إجراء، التحقق من صحة البيانات، تسجيل جميع العمليات الحساسة، تطبيق معدل الطلبات (Rate Limiting).

---

## 9. الأداء والتحسين

- **تحميل كسول:** يتم تحميل الصفحات والمكونات فقط عند الحاجة.
- **تقسيم الكود:** كل صفحة لها ملف JavaScript خاص بها، ويتم تحميله عند التنقل.
- **تحسين الصور:** استخدام SVG للأيقونات، وضغط الصور المرفوعة، واستخدام WebP إذا أمكن.
- **التخزين المؤقت:** استخدام Service Worker لتخزين الموارد الثابتة والبيانات الأساسية.
- **تقليل طلبات API:** دمج البيانات في طلب واحد عند الإمكان، استخدام التخزين المحلي لتجنب الطلبات المتكررة.

---

## 10. التعامل مع الأخطاء

- **على العميل:** عرض رسائل خطأ ودية للمستخدم، تسجيل الأخطاء في وحدة التحكم للتطوير، وإرسال تقارير الأخطاء إلى الخادم (اختياري).
- **على الخادم:** التقاط الاستثناءات، تسجيلها في Logs، وإرجاع استجابة HTTP مناسبة مع رمز خطأ ورسالة مفهومة.

---

## 11. التوثيق والتطوير

- **تعليمات برمجية موثقة:** كل دالة موثقة بتعليق يوضح الغرض، المعاملات، والقيمة المُرجعة.
- **مخططات:** توجد مخططات UML أو تدفق البيانات في مجلد `docs/`.
- **الاختبار:** يتم كتابة اختبارات للوحدات الأساسية باستخدام Jest أو أدوات مماثلة (سيتم تحديدها لاحقاً).

---

## 12. الاعتماديات الخارجية

- لا يوجد إطارات عمل، فقط مكتبات صغيرة (مثل: jsPDF لإنشاء PDF، SheetJS لتصدير Excel) سيتم تحميلها عبر CDN عند الحاجة، مع توثيقها.
